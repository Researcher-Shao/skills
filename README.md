# 中推圈都在教怎么写 Skill，但你可能正在重复 MCP 的错误  
> 写给管理者：Skill 不是工具升级，而是一次组织方式的体检

## TL;DR（一分钟版）
- Skill 不是 prompt，也不是 MCP 的升级版  
- 真正的问题不是“会不会写 Skill”，而是“组织是否愿意改变工作方式”  
- MCP 的失败不是技术失败，而是组织没变  
- Skill 的价值，在于把人的经验与判断标准固化为组织资产  
- AI 时代，工具只是放大器，组织能力才是护城河  

---

## 背景：Skill 热潮与似曾相识的感觉

最近中推圈在疯传“怎么写 Claude Skills”。

这让我想起去年 4 月：  
有个老板拍板搞 MCP（Model Context Protocol），不到百人的团队，不到一年，做出了上百个 MCP。

当时：
- 各团队竞相汇报 MCP
- 每个人都觉得自己做的“很有用”
- 甚至有人以此为荣

结果：
- 真正常用的，不到 10 个

现在 Skill 的风又来了。  
我大胆预测：
- 2 个月内，国内会大热  
- 3 个月后，会出现一波“Skill 没啥用”的声音  

原因只有一个：  
> 大多数公司，又在把“组织变革”的问题，当成“工具采购”的问题来解决。

---

## MCP 的教训：为什么做了上百个却没人用

MCP 刚出来时，团队非常兴奋：

- “终于可以让 AI 连数据库了！”
- “可以直接调用业务服务！”
- “可以读代码库了！”

但现实是：

### 案例一：连 Jira 的 MCP
- 功能：查询任务、状态  
- 结果：几乎没人用  
- 原因：工作流程没变，AI 只是查询工具，还不如直接打开 Jira  

### 案例二：连业务服务的 MCP
- 用了两次就废  
- 原因：还要专门定制 MCP，本质和写接口区别不大  

### 案例三：连代码库的 MCP
- 反馈：“挺好”  
- 实际使用频率：很低  
- 原因：AI 仍然只是辅助工具，而不是工作方式的一部分  

结论很清楚：  
问题不是工具不好，而是我们没想清楚 **工作方式该怎么变**。

---

## Skill 是什么：它更像一面镜子

如果你关注过 AI 工具的演进，大致经历过三个阶段：

- Prompt：把提示词模板复制粘贴给 AI  
- MCP：让 AI 能调用外部工具和数据  
- Skill：把领域知识、流程、判断标准内化成能力  

一个不严谨但好理解的类比：

- Prompt 像是每次给 AI 写使用说明书  
- MCP 让 AI 会用工具，但仍是被动响应  
- Skill 让 AI “知道在什么情况下该怎么做”  

用久了你会发现：  
Skill 不只是工具升级，它更像一面镜子，照出了组织还停留在前 AI 时代的运作方式。

---

## 技术为什么演进到 Skill

通用大模型已经很强，但有一个根本问题：

> 它对你的业务一无所知。

它不知道：
- 你们用什么技术栈  
- 为什么当初选了这个架构  
- 哪些坑已经踩过  
- 哪些地方不能轻易动  

过去的解决方案：
- Prompt 模板：累，而且不可持续  
- RAG：只是参考资料，不是判断能力  

Skill 的本质不同在于：
- 不是“提醒” AI  
- 而是让 AI 内化判断标准和工作方式  

这是技术发展的必然结果，而不是某家公司的偶然创新。

---

## 为什么同样的 AI，效果差距巨大

在 AI coding 推广中，常见一个现象：

- 有些人用 AI 写的代码：结构清晰、规范完整，稍微改改就能合并  
- 有些人用 AI 写的代码：一眼 AI 味，命名随意、边界不全，Review 要来回好几轮  

这不是工具熟练度的问题，而是：

> AI 是能力放大器。

你的 prompt 背后，是你的工程理解、质量判断和经验积累。  
AI 不会替你补上这些。

结果是：
- 高手：效率翻倍，质量还能保持  
- 新手：速度快了，但质量很难保证  

Skill 的意义在于：  
把高手的经验和判断标准，变成组织的默认能力。

---

## Skill 的本质：从人驱动到 AI 自驱动

传统模式：
新人写代码 → 高手 Code Review → 修改 → 再 Review  
高手累，新人挫败，扩展性很差。

Skill 模式：
高手定义标准 → Skill 固化 → AI 按标准执行 → 人做决策和调整  

角色发生了变化：
- 高手从“执行者”变成“标准制定者”  
- 新人从“反复试错”变成“基于标准决策”  

这是从“人驱动执行”到“AI 自驱动执行”的转变。

---

## Skill 暴露的组织脆弱性

一个架构师离职，会带走什么？

- 为什么当初选这个方案  
- 哪些地方是历史妥协  
- 遇到新问题该怎么判断  

这些东西很难写进文档，但决定了系统未来。

Skill 能做的不是替代人，而是：
- 固化判断逻辑  
- 保留设计思路  
- 降低组织对个人的依赖  

AI Native 组织的核心特征之一：  
**知识资产不再依附于个人。**

---

## Skill 背后的更大图景：组织必须重构

三个连锁趋势：

1. 可 SOP 化的岗位会消失  
如果你的工作能写成流程，AI 迟早能做。

2. 角色边界会模糊  
PM 写代码，工程师验证产品，端到端能力更重要。

3. 工作流核心变成 AI 的自主性  
人定目标和标准 → AI 自主规划和执行 → 人做决策  

这不是工具问题，而是价值创造方式的改变。

---

## 为什么必须一把手站出来

因为这涉及：
- 工作流程重构  
- 考核方式调整  
- 组织结构变化  
- 试错成本承担  

这些事情，不可能由某个部门推动。  
必须一把手明确表态、亲自推动。

---

## 如果你现在就想开始

最低风险的方式：

1. 选一个 3–5 人的小团队  
2. 定一个具体、可衡量的小目标  
3. 给 1 个月探索和试错  
4. 每周复盘：什么 work，什么不 work  
5. 把有效方法固化成 Skill、文档或团队共识  

一个月后，你会有真实数据，而不是猜测。

---

## 不要这样做

- 不要一上来就全员铺开  
- 不要把 Skill 数量当 KPI  
- 不要期待立竿见影  
- 不要只买工具不改流程  

---

## 我的预判

接下来你会看到：
- Skill 教程刷屏  
- 公司要求“每个团队做几个 Skill”  
- KPI 变成 Skill 数量  

然后你会听到：
- “Skill 也没那么神奇”
- “做了一堆但没人用”

这不是 Skill 的问题，而是组织没变。

## 后记：这篇文章本身就是 Skill 的实验

这篇文章的写作方式：

- 人负责：观点、判断、方向  
- AI 负责：整理、结构化、补充  
- 工具负责：连接和沉淀（Obsidian、Claude Code）  

这就是我理解的 AI Native 工作方式。

如果你的组织还没开始这样工作，  
不妨从一个小场景试试看。  
效果，会比读十篇文章更直接。
